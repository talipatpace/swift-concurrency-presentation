Why Concurrency?

The problems we've always had

Programs are linear. Apps are not.

We write code top-to-bottom. But real apps fetch data, respond to touches, and animate — all at the same time.

let a = 2
let b = 3
let sum = a + b
print(sum)        // simple, predictable, linear

Classical pitfalls

Deadlocks

Thread A waits for B, B waits for A. Neither proceeds — forever.

Race Conditions

Multiple threads write the same data without coordination. Silent corruption.

Starvation

A thread waits for resources it never receives. The Dining Philosophers problem.

Swift's concurrency model addresses all of these at the language level. Many violations become compile-time errors.

The pyramid of doom

func fetchProfile() {
    authenticateUser { success, error in
        if success {
            fetchUserInfo { userInfo, error in
                DispatchQueue.main.async {
                    if let userInfo = userInfo {
                        fetchFollowers { followers, error in
                            DispatchQueue.main.async {
                                // Finally update UI...
                            }
                        }
                    }
                }
            }
        }
    }
}

Nested callbacks break readability and make error handling a nightmare.

async / await

Linear code that can pause

The async keyword

Mark a function async to signal it may suspend. Place it after parameters, before throws.

func fetchUserInfo() async throws -> UserInfo {
    let (data, _) = try await URLSession.shared.data(from: url)
    return try JSONDecoder().decode(UserInfo.self, from: data)
}

let user = try await fetchUserInfo()

Key insight: async means "can pause" — it does not mean "runs in the background."

The await keyword

Marks a potential suspension point. Execution pauses here and may resume on a different thread.

func authenticateAndFetch() async {
    print("Fetching user info")        // runs immediately
    let user = try await fetchUserInfo()
    print("User info fetched")          // runs after — maybe different thread
}
Thread 1 → print → await (suspend) → thread freed
···
Thread 2 → resume → print → continue

Suspension is never implicit. You always see await in source code.

Before and after

Closure-based 
userApi.fetchUserInfo { info, err in
    DispatchQueue.main.async {
        if let info = info {
            userApi.fetchFollowers { f, e in
                DispatchQueue.main.async {
                    self.followers = f
                }
            }
        }
    }
}
async/await 
@MainActor
func loadProfile() async throws {
    let info = try await userApi
        .fetchUserInfo()
    let followers = try await userApi
        .fetchFollowers()
    self.userInfo = info
    self.followers = followers
}

Where async code can run

  • Inside other async functions, methods, or computed properties
  • In @main struct/class static func main()
  • Inside a Task { } block — the bridge from sync to async
  • In SwiftUI's .task { } view modifier
// Bridging: calling async from synchronous code
func viewDidLoad() {
    Task {
        let profile = try await fetchUserInfo()
        self.nameLabel.text = profile.name
    }
}

Structured Concurrency

Parallel work with guardrails

async let

Run independent work in parallel instead of waiting one-by-one:

Sequential — slow 
let a = await download("a.jpg")
let b = await download("b.jpg")
let c = await download("c.jpg")
// time ≈ a + b + c
Parallel — fast 
async let a = download("a.jpg")
async let b = download("b.jpg")
async let c = download("c.jpg")
let photos = await [a, b, c]
// time ≈ max(a, b, c)

Task Groups dynamic parallelism

When the number of tasks isn't known at compile time:

let photos = await withThrowingTaskGroup(of: Data.self) { group in
    let names = await listPhotos(inGallery: "Summer Vacation")

    for name in names {
        group.addTask {
            return try await downloadPhoto(named: name)
        }
    }

    var results: [Data] = []
    for try await photo in group {
        results.append(photo)
    }
    return results
}

Parent cannot forget child tasks. Cancel parent → all children cancel automatically.

The task tree

Parent Task
├── Child (async let)
├── Child (async let)
└── Task Group
    ├── addTask { ··· }
    ├── addTask { ··· }
    └── addTask { ··· }
  • Cancellation propagates down — cancel parent, all children cancel
  • Priority escalates up — high-priority child elevates parent
  • Parent always waits for children before returning

Cooperative cancellation

Tasks must check for cancellation and respond. Never forced.

// Throw immediately if cancelled
try Task.checkCancellation()

// Check and return partial results
if Task.isCancelled {
    return partialResults
}

// Don't start new work after cancellation
let added = group.addTaskUnlessCancelled {
    guard !Task.isCancelled else { return nil }
    return try await downloadPhoto(named: name)
}
guard added else { break }

Unstructured Concurrency

When structure doesn't fit

Task { }

// Inherits actor, priority, and task-local values
let handle = Task {
    return try await fetchUserProfile()
}

let profile = try await handle.value   // await result later
handle.cancel()                         // or cancel it
Task { }

Inherits actor isolation, priority, and task-local state from the caller.

Task.detached { }

No inherited context. Truly independent. Use sparingly.

SwiftUI's .task modifier

struct ProfileView: View {
    @State private var avatar: Image?

    var body: some View {
        AvatarView(avatar)
            .task {
                avatar = await downloadAvatar()
            }
            .task(id: userID) {
                avatar = await downloadAvatar(for: userID)
            }
    }
}

Starts on appear, cancels on disappear. Prefer this over bare Task { } in SwiftUI.

Actors

Shared mutable state, made safe

What is an actor?

A reference type that serializes access to its mutable state. Only one task at a time.

actor TemperatureLogger {
    let label: String
    var measurements: [Int]
    private(set) var max: Int

    func update(with measurement: Int) {
        measurements.append(measurement)
        if measurement > max {
            max = measurement   // no await needed inside the actor
        }
    }
}

Isolation in practice

let logger = TemperatureLogger(label: "Outdoors", measurement: 25)

// From outside — requires await
print(await logger.max)           // ✓
await logger.update(with: 30)     // ✓

// Without await — compile error
print(logger.max)                 // ✗ actor-isolated property

Think of an actor as a private office. One person works inside at a time. You can't barge in — you knock (await) and wait your turn.

Actor reentrancy

State can change across suspension points:

actor VideogameLibrary {
    var videogames: [Videogame] = []

    func addGamesAndReport(newGames: [Videogame]) async {
        let countBefore = videogames.count
        try await Task.sleep(for: .seconds(1))       // ⚠️ suspension!
        videogames.append(contentsOf: newGames)
        // countBefore may be WRONG — another task modified state
        print("Added \(videogames.count - countBefore) games")
    }
}

Rule: Never assume state is unchanged after an await inside an actor.

nonisolated

Opt out of isolation for methods that only read immutable state:

actor VideogameLibrary {
    let owner: Owner    // immutable — safe everywhere

    nonisolated func fetchOwnerInfo() -> String {
        "\(owner.name) loves \(owner.favoriteGenre)"
    }
}

// No await needed
let info = library.fetchOwnerInfo()

@MainActor

Protecting UI state

Main thread, made explicit

@MainActor
class ProfileViewModel: ObservableObject {
    @Published var userName: String = ""
    @Published var avatar: UIImage?

    func loadProfile() async throws {
        let profile = try await api.fetchProfile()
        self.userName = profile.name    // safe — on MainActor
    }
}

// Calling from background needs await
func downloadAndShow(named name: String) async {
    let photo = await downloadPhoto(named: name)
    await show(photo)    // hop to MainActor
}

Approachable Concurrency Swift 6.2+

New default in Xcode 26: everything starts on @MainActor.

// Default build settings in Swift 6.2+
// SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor
// SWIFT_APPROACHABLE_CONCURRENCY = YES
  • Functions stay on the caller's actor by default
  • Use @concurrent to explicitly opt into background execution
@concurrent
func processImage(_ data: Data) -> UIImage {
    // CPU-intensive work, explicitly off the main thread
}

Sendable Types

Safe data across boundaries

What is Sendable?

A marker protocol: "this type is safe to pass between concurrency domains."

Value types

Structs/enums with sendable properties are implicitly Sendable.

Actors

Sendable by design. Their purpose is safe shared state.

Classes

Not Sendable by default. Reference semantics = shared mutation risk.

Sendable in code

// Implicitly Sendable — all properties are value types
struct TemperatureReading {
    var measurement: Int
    var timestamp: Date
}

// Explicitly Sendable — must be final, immutable properties
final class Config: Sendable {
    let apiKey: String
    let timeout: TimeInterval
}

Escape hatches

// @unchecked Sendable — "trust me, I handle thread safety"
class DatabaseConnection: @unchecked Sendable {
    private let lock = NSLock()
    private var _cache: [String: Data] = [:]

    func read(_ key: String) -> Data? {
        lock.lock()
        defer { lock.unlock() }
        return _cache[key]
    }
}

Use @unchecked Sendable sparingly. It's a promise to the compiler — not a guarantee.

Continuations

Bridging old APIs to async/await

Wrapping callbacks

func fetchUserInfo() async throws -> UserInfo {
    try await withCheckedThrowingContinuation { continuation in
        legacyFetchUserInfo { userInfo, error in
            if let error = error {
                continuation.resume(throwing: error)
            } else if let userInfo = userInfo {
                continuation.resume(returning: userInfo)
            }
        }
    }
}

let user = try await fetchUserInfo()

Checked vs. unsafe

Checked variants

withCheckedContinuation
withCheckedThrowingContinuation
Runtime checks — resume exactly once.

Unsafe variants

withUnsafeContinuation
withUnsafeThrowingContinuation
No checks — faster. Use after testing.

A continuation must be resumed exactly once. Zero = leaked task. Twice = undefined behavior.

Wrapping delegates

func requestBiometricUnlock() async throws -> Bool {
    try await withCheckedThrowingContinuation { continuation in
        LAContext().evaluatePolicy(
            .deviceOwnerAuthenticationWithBiometrics,
            localizedReason: "To access your data"
        ) { success, error in
            if let error { continuation.resume(throwing: error) }
            else { continuation.resume(returning: success) }
        }
    }
}

A continuation must be resumed exactly once. Zero = leaked task. Twice = undefined behavior.

AsyncSequence

Values over time

Iterating asynchronously

// Read lines as they arrive
let handle = FileHandle.standardInput
for try await line in handle.bytes.lines {
    print(line)
}

// URLSession bytes
let (bytes, _) = try await URLSession.shared.bytes(from: url)
for try await byte in bytes {
    process(byte)
}

for try await suspends at each iteration, waiting for the next element.

AsyncStream

let locations = AsyncStream<CLLocation> { continuation in
    let delegate = LocationDelegate()
    delegate.onLocationUpdate = { continuation.yield($0) }
    delegate.onFinish = { continuation.finish() }
    continuation.onTermination = { _ in delegate.stopUpdating() }
    delegate.startUpdating()
}

for await location in locations {
    updateMap(location)
}

Best Practices

Patterns that scale

Common mistakes

  • Confusing async with background — async means "can pause," not "runs off main"
  • Over-engineering with actors — most apps only need @MainActor
  • Blocking the cooperative pool — never DispatchSemaphore.wait() in async
  • Bare Task { } — prefer .task or TaskGroup for lifecycle management
  • Assumptions across await — actor state changes during suspension
  • Unnecessary Sendable — only needed when crossing isolation boundaries

Source: fuckingapproachableswiftconcurrency.com

Do's and don'ts

Do
  • Keep actors small and focused
  • Make actor operations atomic
  • async let for known parallel work
  • TaskGroup for dynamic parallelism
  • .task modifier in SwiftUI
  • @concurrent for CPU work (6.2+)
Don't
  • Block the thread pool with sync waits
  • Ignore Task.isCancelled
  • Assume state after await
  • Pass mutable ref types across boundaries
  • Task.detached unless truly needed
  • @unchecked Sendable to silence warnings
